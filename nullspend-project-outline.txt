# NullSpend — Project Brief

## One-line summary

NullSpend is a lightweight approval layer for AI agent actions.

It sits between an agent and any risky side effect, such as sending an email, making an HTTP POST request, writing to a database, running a shell command, or deleting a file. Instead of letting the action execute immediately, NullSpend turns it into a proposed action, shows it in an approval inbox, waits for approval or rejection, and then either executes or blocks the action.

The goal for v1 is not to build a giant governance platform. The goal is to build the smallest useful developer tool that proves this loop works.

---

## Product framing

### Problem

AI agents are getting better at reasoning and tool use, but developers still do not trust them to freely touch real systems.

Examples:
- sending outbound emails
- updating CRM records
- charging cards
- deleting files
- executing shell commands
- posting to APIs
- writing to databases

Today, developers usually handle this in ad hoc ways:
- console prompts
- commented-out tool calls
- custom approval logic
- framework-specific middleware
- Slack messages and manual review hacks

That works for one-off prototypes, but it becomes messy quickly.

### Solution

NullSpend provides:
- a dead-simple approval inbox
- a tiny SDK/wrapper
- a backend that stores pending actions
- a clean pause-and-resume approval flow
- logs and action history
- optional signed receipts later

### Core user promise

Wrap risky agent actions in minutes, preview them in a dashboard, approve or reject them, and keep a clean log of what happened.

---

## v1 product scope

### Core loop

1. An agent wants to perform a risky action.
2. The developer has wrapped that action using NullSpend.
3. NullSpend creates a pending action in the backend.
4. The action appears in the dashboard.
5. A human approves or rejects it.
6. If approved, the original function executes.
7. If rejected, the function does not execute.
8. The result is logged.

### v1 supported action types

Start small. Only support a few obvious risky actions:
- `send_email`
- `http_post`
- `http_delete`
- `shell_command`
- `db_write`
- `file_write`
- `file_delete`

Do not try to support every framework and every tool on day one.

## Mental model

NullSpend is not the agent and not the tool.

NullSpend is the checkpoint between the agent and the real world.

Without NullSpend:

```text
Agent → risky action executes immediately
```

With NullSpend:

```text
Agent → proposed action → approval decision → execute or block
```

## Core abstractions

### 1. Proposed Action

A proposed action is a risky action that has not executed yet.

Example shape:

```ts
type ActionStatus =
  | "pending"
  | "approved"
  | "rejected"
  | "executing"
  | "executed"
  | "failed"
  | "expired"

type ProposedAction = {
  id: string
  agentId: string
  actionType: string
  status: ActionStatus
  payload: Record<string, unknown>
  metadata?: Record<string, unknown>
  createdAt: string
  approvedAt?: string | null
  rejectedAt?: string | null
  executedAt?: string | null
  approvedBy?: string | null
  rejectedBy?: string | null
  result?: Record<string, unknown> | null
  errorMessage?: string | null
}
```

### 2. Risky Action Wrapper

A wrapper is a helper that intercepts a function before it performs a side effect.

Examples:

* wrap `sendEmail()`
* wrap `fetch()` for POST/DELETE
* wrap `db.insert()`
* wrap `subprocess.run()`
* wrap `fs.writeFile()`

### 3. Approval Inbox

A dashboard that shows pending actions and allows:

* approve
* reject
* inspect payload
* inspect logs
* later: approve once, auto-approve, rules, receipts

Use Supabase Auth for user accounts.

You may optionally mirror lightweight user profile fields in your own table later.

Useful if you want a real event timeline.

Example event types:

* `created`
* `approved`

This is where your original signing idea can plug in.

Do **not** build receipts first. Add them after the core loop works.

## API design

Keep the API dead simple.

### Create action

`POST /api/actions`

Request:

```json
{
  "agentId": "sales-agent-1",
  "actionType": "send_email",
  "payload": {
    "to": "sarah@example.com",
    "subject": "Follow up",
    "body": "Hi Sarah..."
  },
  "metadata": {
    "environment": "dev",
    "sourceFramework": "custom-python"
  }
}
```

Response:

```json
{
  "id": "act_123",
  "status": "pending"
}
```

### Get action by id

`GET /api/actions/:id`

Response:

```json
{
  "id": "act_123",
  "status": "approved",
  "approvedBy": "chris@example.com"
}
```

### Approve action

`POST /api/actions/:id/approve`

Request:

```json
{
  "approvedBy": "user_123"
}
```

### Reject action

`POST /api/actions/:id/reject`

Request:

```json
{
  "rejectedBy": "user_123",
  "reason": "Looks incorrect"
}
```

### List actions

`GET /api/actions?status=pending`

This is useful for the inbox, but it should stay secondary to the core create, decision, and result flow.

### Mark execution result

`POST /api/actions/:id/result`

Request:

```json
{
  "status": "executed",
  "result": {
    "message": "Email sent successfully"
  }
}
```

Or:

```json
{
  "status": "failed",
  "errorMessage": "SMTP timeout"
}
```

## State machine

Use a clear action state machine.

```text
pending
  ├─ approve → approved
  ├─ reject  → rejected
  └─ timeout → expired

approved
  └─ begin execution → executing

executing
  ├─ success → executed
  └─ error   → failed
```

This should be explicit in code.

## SDK design

### Goal

The SDK should be tiny.

It should not try to be a full framework.

### Core helper

A simple helper could look like this:

```ts
await proposeAndWait({
  agentId: "sales-agent-1",
  actionType: "http_post",
  payload: { url, body },
  execute: async () => {
    return await fetch(url, {
      method: "POST",
      body: JSON.stringify(body),
    })
  }
})
```

### TypeScript sketch

```ts
type ProposeOptions<T> = {
  agentId: string
  actionType: string
  payload: Record<string, unknown>
  metadata?: Record<string, unknown>
  execute: () => Promise<T>
  pollIntervalMs?: number
  timeoutMs?: number
}

async function proposeAndWait<T>(opts: ProposeOptions<T>): Promise<T> {
  const action = await createAction({
    agentId: opts.agentId,
    actionType: opts.actionType,
    payload: opts.payload,
    metadata: opts.metadata,
  })

  const decision = await waitForDecision(action.id, {
    pollIntervalMs: opts.pollIntervalMs ?? 2000,
    timeoutMs: opts.timeoutMs ?? 5 * 60 * 1000,
  })

  if (decision.status !== "approved") {
    throw new Error(`Action ${decision.status}`)
  }

  await markExecutionStarted(action.id)

  try {
    const result = await opts.execute()
    await markExecutionSucceeded(action.id, result)
    return result
  } catch (err) {
    await markExecutionFailed(action.id, String(err))
    throw err
  }
}
```

### Python sketch

```python
def propose_and_wait(agent_id, action_type, payload, execute, metadata=None):
    action = create_action(
        agent_id=agent_id,
        action_type=action_type,
        payload=payload,
        metadata=metadata or {}
    )

    decision = wait_for_decision(action["id"])

    if decision["status"] != "approved":
        raise Exception(f'Action {decision["status"]}')

    mark_execution_started(action["id"])

    try:
        result = execute()
        mark_execution_succeeded(action["id"], result)
        return result
    except Exception as e:
        mark_execution_failed(action["id"], str(e))
        raise
```

### Decorator idea in Python

```python
def require_approval(action_type):
    def decorator(fn):
        def wrapper(*args, **kwargs):
            return propose_and_wait(
                agent_id="default-agent",
                action_type=action_type,
                payload={"args": args, "kwargs": kwargs},
                execute=lambda: fn(*args, **kwargs),
            )
        return wrapper
    return decorator
```

Usage:

```python
@require_approval("send_email")
def send_email(to, subject, body):
    ...
```

## Waiting strategy

### v1: polling

Use polling first.

It is the simplest and works everywhere.

Example:

* SDK creates action
* SDK polls `GET /api/actions/:id` every 2 seconds
* once status changes, continue or fail

### Do not start with

* websockets
* realtime streaming
* complex resumable workflows
* event buses

Those can come later.

## MVP success criteria

v1 is successful if all of these are true:

1. A developer can wrap a risky action.
2. That action appears in a web inbox.
3. A human can approve or reject it.
4. Approval causes the original action to execute.
5. Rejection blocks the action.
6. The action history is stored and visible.

That is enough.

“Build an inbox page that fetches pending actions from the API and renders them in a simple table with Approve and Reject buttons.”
